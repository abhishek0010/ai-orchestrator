### [SylphxAI/pdf-reader-mcp](https://github.com/SylphxAI/pdf-reader-mcp) ⭐ 809 · TypeScript · Updated: 0 days ago

**What it does:** Provides a fast, evidence‑first PDF “MCP” server that returns a richly‑structured document twin (text, layout, OCR, trust reports, citations, etc.) for LLM agents.

**Problem it solves for ai-orchestrator:** The orchestrator currently lacks a native way for agents to ingest PDFs while preserving provenance (e.g. `AgentLoop.ts` may need a PDF → markdown conversion, but we only have plain‑text tools). Adding a PDF‑twin tool would let agents request source evidence, citation IDs, and visual crops without writing custom extraction code.

**How to integrate:**
- **File:** `src/core/ToolRunner.ts`
- **Change:**
  1. Add a new `case "read_pdf":` branch in the `execute()` switch that builds a request payload for the PDF‑Reader MCP CLI (`npx @sylphx/pdf-reader-mcp`) and captures its JSON response.
  2. Parse the returned structure and write a context file (e.g. `.claude/context/pdf_twin_<hash>.md`) containing the Markdown output and a concise evidence index.
  3. Return a `ToolResult` object with `{ ok: true, outputPath: "...", evidenceIds: [...] }` that downstream agents can reference.

**Effort:** Medium (days) – requires adding the new tool case, wiring CLI invocation, and defining the result schema, but leverages the existing `ToolRunner` pattern.

**Risk:** Low – the PDF‑Reader MCP is a standalone CLI; integration is a simple subprocess call with no impact on existing orchestration logic.

**Verdict:** **ADOPT** ✅ – Adding a “read_pdf” tool gives ai‑orchestrator agents immediate, trustworthy access to PDF sources, complementing the current text‑only workflow and unlocking evidence‑based reasoning.

### [zzet/gortex](https://github.com/zzet/gortex) ⭐ 823 · Go · Updated: 0 days ago

**Problem it solves for ai‑orchestrator:** `ContextPruner.ts` currently relies on plain‑text pruning and an optional Headroom CLI; it lacks a native, low‑overhead binary serializer that can shrink context payloads before they hit the LLM.

**How to integrate:**
- **File:** `src/core/HeadroomBridge.ts`
- **Change:**
  1. Add a small TypeScript implementation of the GCX1 encoder/decoder (port the Go logic that packs a JSON object into a length‑prefixed binary buffer).
  2. In `compressWithHeadroom`, after the Headroom CLI fallback, call `gcx1Compress(jsonString)` and return the binary string (base‑64‑encoded) when the encoder is available.

**Effort:** Quick (hours)
**Risk:** Low – the encoder is pure‑JS/TS, no external dependencies, and only adds a fallback path.
**Verdict: ADOPT** ✅ – brings measurable token savings without breaking existing tooling.

### [zzet/gortex](https://github.com/zzet/gortex) ⭐ 823 · Go · Updated: 0 days ago — 2

**What it does:** Builds an in‑process, persistent symbol graph (functions, classes, call‑chains) that enables O(1) look‑ups of “where is X defined?” across many repositories.

**Problem it solves for ai‑orchestrator:** `ToolRunner.ts`’s `read_file` and `search_files` tools always read whole files; this wastes tokens when agents only need a single symbol’s definition.

**How to integrate:**
- **File:** `src/core/ToolRunner.ts`
- **Change:**
  1. Add a new private method `querySymbolGraph(symbol:string): string | null` that invokes a new Bash script `scripts/build-symbol-index.sh` (ported from Gortex’s index builder) to query a SQLite DB created at daemon start.
  2. Extend the `execute` switch‑case for `read_file` to first call `querySymbolGraph`; if a result is found, return the snippet instead of the full file.

**Effort:** Medium (days) – requires adding a lightweight index builder script and a SQLite lookup wrapper.
**Risk:** Medium – the index must stay in sync with repo changes; we’ll use the existing file‑watcher events in `AgentLoop.ts` to trigger a rebuild.
**Verdict: ADOPT** ✅ – dramatically cuts context size for symbol‑centric queries, matching Gortex’s core value proposition.

### [zzet/gortex](https://github.com/zzet/gortex) ⭐ 823 · Go · Updated: 0 days ago — 3

**What it does:** Offers a “preview_edit / simulate_chain” feature that computes the impact of a proposed `WorkspaceEdit` without touching the filesystem.

**Problem it solves for ai‑orchestrator:** `DiffCompressor.ts` only compresses diffs after they are written; it cannot tell the planner whether a change would cause a large cascade before committing.

**How to integrate (WATCH):**
- **Pattern to study:** Gortex’s speculative execution pipeline (`preview_edit` command) – it parses the pending edit, runs the same graph resolution logic on a shadow copy of the index, and returns a “blast‑radius” report.
- **When to port:** If `ContextPruner` or `GoalQueue` starts to experience frequent “edit‑caused‑retries” where the planner repeatedly backs out of changes after a review, adopting a speculative check will prevent wasted LLM calls.

**Effort:** Large (weeks) – requires building a shadow graph layer and exposing it to the TS daemon.
**Risk:** High – introduces complex state management and may affect performance if not carefully cached.
**Verdict: WATCH** 👀 – keep an eye on edit‑retry patterns; implement when they become a bottleneck.
