### [zhnt/loushang](https://github.com/zhnt/loushang) ⭐ 631 · Python · Updated: 0 days ago

**What it does:** Provides a Python SDK that routes LLM calls to multiple providers, persisting session state and tooling information.

**Problem it solves for ai-orchestrator:** `src/core/LlmHttpClient.ts` currently hard‑codes provider selection logic via `llm-config.json`; adding a reusable provider‑registry and session‑persistence layer would simplify model routing and enable resume‑able conversations (similar to Loushang’s `Session` object).

**How to integrate:**
- **File:** `src/core/LlmHttpClient.ts`
- **Change:**
  1. Import a new `ProviderRegistry` interface (mirroring Loushang’s `ProviderCatalog`) and load it from `src/ai-sdk/provider-catalog.ts` (new file).
  2. Replace the current `resolveModelForRole` logic with a call to `ProviderRegistry.getModelForRole(role)` which returns `{endpoint, model}` based on the catalog, falling back to the existing `llm-config.json` if the catalog is empty.
  3. After each successful LLM call, write the request/response pair into a persistent `sessions/<role>/<timestamp>.json` file (creating a simple session store like Loushang’s `Session`), enabling later resume or audit.

**Effort:** Medium (≈2 days)

**Risk:** Low — only adds a thin abstraction layer and file writes; existing code paths remain unchanged if the catalog is empty.

**Verdict:** **ADOPT** ✅ – The provider‑registry and session persistence pattern from Loushang can be safely ported to improve model routing flexibility and auditability in ai‑orchestrator.
