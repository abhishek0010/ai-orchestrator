## New Projects Analyzed

### [OpenSwarm](https://github.com/unohee/OpenSwarm) ⭐ 809
- **What it does**: Autonomous AI dev team orchestrator built on Claude Code CLI. Picks up Linear issues, runs Worker→Reviewer pipelines, reports to Discord, and retains per-repo long-term memory via LanceDB. Supports Claude, GPT, Codex, OpenRouter, and local models (Ollama/LMStudio). Includes a TUI with Chat/Projects/Tasks/Stuck/Logs tabs, cost tracking, and a code-smell "BS Detector."
- **Why relevant to ai-orchestrator**: Nearly identical mission. TypeScript codebase with the same multi-agent pipeline pattern (planner→coder→reviewer→commit). Also integrates with Ollama and uses Claude Code CLI as the execution backend.
- **Ideas to adopt**:
  - **LanceDB for the knowledge base**: Replace `knowledge/embeddings.jsonl` with LanceDB. OpenSwarm stores per-repo task outcomes as vector embeddings and recalls them into future prompts automatically. LanceDB is faster and supports semantic search out of the box.
  - **Hybrid frontier/lightweight execution**: OpenSwarm's hybrid mode has a frontier model (Opus) diagnose the problem read-only, then a lightweight model (Haiku) implement with a verification loop. Resolved 3/3 SWE-bench Lite instances lightweight-only had failed. Apply this to our planner→coder handoff: planner runs Opus for decomposition, coder runs Sonnet/Haiku for implementation.
  - **`--pipeline` flag with roles**: OpenSwarm's `exec --pipeline` runs Worker + Reviewer + Tester + Documenter. Our `/implement` slash command could similarly compose all four roles in sequence rather than hard-coding only planner→coder→reviewer→commit.
  - **Code Registry with drift detection**: `openswarm check --scan` builds a per-entity registry and `--bs` flags anti-patterns. Our performance-monitor agent (`agents/performance-monitor/`) could add a similar registry scan on commit.
  - **Cost tracking in the TUI**: OpenSwarm's status bar shows cumulative cost per session. Our context-manager agent could track and surface token/cost estimates.

### [attune-ai](https://github.com/Smart-AI-Memory/attune-ai) ⭐ 6
- **What it does**: Spec-driven development platform for Claude Code. 20 workflows (17 multi-stage), 17 auto-triggering skills, 41 MCP tools. Dispatches parallel teams of 2–6 subagents with model assignment by complexity (Opus/Sonnet/Haiku). RAG-grounded generation at 0.996 mean per-claim faithfulness. Cross-session memory via Stop/SessionStart/UserPromptSubmit/PreToolUse hooks. On-demand `/recall <topic>` command.
- **Why relevant to ai-orchestrator**: Directly tackles the same skill/workflow architecture we have, with mature solutions to our known gaps: cross-session memory, model tier routing, and RAG-grounded knowledge retrieval.
- **Ideas to adopt**:
  - **Cross-session memory hooks**: attune-ai's `Stop` hook extracts decisions and bugs from the completed session (using local LLM when available, heuristic fallback) and writes them to a memory store; `SessionStart` hook surfaces the most recent findings. Wire this into our `context-manager` agent — our current `knowledge/outcomes.jsonl` would be the persistence layer.
  - **Tiered model assignment**: Opus for deep reasoning (planner, architect, security-auditor), Sonnet for general analysis (coder, reviewer), Haiku for fast scanning (error-coordinator, context-manager). Currently there is no per-agent model configuration. Add `model` field to agent configs in `agents/*/`.
  - **PreToolUse hook for lesson injection**: At the exact tool call moment (e.g., before a file Edit), surface relevant engineering lessons from `knowledge/outcomes.jsonl`. This is more targeted than system-prompt injection.
  - **Socratic clarification before execution**: attune-ai's workflows "ask before act" — e.g., `spec` workflow brainstorms, plans, then executes. Our `planner` agent could add a clarification step before generating the implementation plan.
  - **Parallel subagent dispatch**: attune-ai's `security-audit` dispatches vuln-scanner + secret-detector + auth-reviewer + remediation-planner in parallel. Our `security-auditor` agent currently runs sequentially — parallelizing would cut audit time significantly.

### [review-flow](https://github.com/DGouron/review-flow) ⭐ 38
- **What it does**: Automated AI code reviews via Claude Code CLI, triggered by GitLab/GitHub webhooks. Configurable sets of specialized audit agents (Clean Architecture, SOLID, Testing, DDD, Security, Quality). Built-in MCP server with typed tools for review progress tracking. WebSocket dashboard with real-time per-agent progress, per-developer performance metrics, and follow-up loop when fixes are pushed.
- **Why relevant to ai-orchestrator**: Our `reviewer` agent does a single-pass review. review-flow shows a production pattern for multi-agent parallel review with typed progress reporting and developer analytics.
- **Ideas to adopt**:
  - **MCP-based review progress tools**: Instead of parsing text output from the reviewer, define typed MCP tools (`start_agent`, `complete_agent`, `set_phase`, `add_action`) so the orchestrator tracks review state structurally. Modify `agents/reviewer/` to call these tools.
  - **Configurable review agent roster**: review-flow's `agents` config array lets teams add/remove audit focuses (security, testing, architecture). Our `/review` skill could accept a config for which specialist sub-agents to run.
  - **Per-developer metrics in knowledge base**: review-flow tracks Quality, Responsiveness, Code Volume, and Iteration rate per developer. Our `knowledge/outcomes.jsonl` schema could add a `contributor` dimension to surface patterns over time.
  - **Concurrent review queue with deduplication**: review-flow uses p-queue with deduplication (prevents re-reviewing the same PR within a time window) and a memory guard (auto-kills if RSS > 4 GB). Our error-coordinator agent could integrate similar concurrency limits.

### [attune-ai / basou](https://github.com/basou-dev/basou) ⭐ 1
- **What it does**: Provenance layer for AI agent sessions (wraps Claude Code CLI). Records every observable event — command runs, file changes, decisions, approvals, git snapshots — as JSONL in `.basou/sessions/<id>/events.jsonl`. Generates `.basou/handoff.md` and `.basou/decisions.md` from the event log. Localhost-only web UI for browsing sessions. Read-only SDK at `@basou/sdk`.
- **Why relevant to ai-orchestrator**: Our agents currently lack a structured audit trail. Basou's JSONL event log approach directly informs how our context-manager should record session provenance.
- **Ideas to adopt**:
  - **JSONL provenance log per session**: Add a `.ai-orchestrator/sessions/<id>/events.jsonl` alongside the existing `knowledge/outcomes.jsonl`. Each orchestrator action (agent dispatch, tool call, approval) appends a typed event. This enables replay and audit without changing the current flow.
  - **Auto-generated handoff.md**: After `/commit`, generate a `handoff.md` summarizing what changed, decisions made, and open questions. The `context-manager` agent could own this. Currently the commit message is the only artifact.

## Top Recommendations for ai-orchestrator

1. **Replace `knowledge/embeddings.jsonl` with LanceDB** — OpenSwarm proves this is production-ready for per-repo semantic recall. Modify `context-manager` agent to write/query LanceDB instead of flat JSONL. Enables semantic search over past task outcomes rather than exact-match lookup.

2. **Add cost-aware tiered model routing** — Both OpenSwarm and attune-ai use Opus/Sonnet/Haiku split by task complexity. Add a `model` field to each agent's config (e.g., `agents/planner/`, `agents/coder/`) and a central routing table. Planner + security-auditor → Opus; coder + reviewer → Sonnet; error-coordinator + context-manager → Haiku. Estimated 60-70% cost reduction with equivalent quality.

3. **Wire cross-session memory via Stop/SessionStart hooks** — attune-ai's pattern of extracting session findings on Stop and surfacing them on SessionStart directly addresses our lack of memory continuity. The `context-manager` agent should own a Stop hook that writes a structured summary to `knowledge/outcomes.jsonl`, and a SessionStart hook that loads the last N findings into the system prompt.

4. **Typed MCP tools for reviewer progress** (from review-flow) — Replace text-output parsing in `agents/reviewer/` with a small MCP server exposing `start_agent`, `complete_agent`, and `add_action` tools. Gives the orchestrator real-time visibility into which review checks are passing/failing.

5. **JSONL session provenance log + handoff.md** (from basou) — Add event logging in `context-manager` so every agent dispatch, decision, and approval is recorded to `.ai-orchestrator/sessions/<id>/events.jsonl`. Auto-generate a `handoff.md` at the end of each `/commit` run for async collaboration.
