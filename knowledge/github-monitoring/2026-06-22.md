**Problem it solves for ai-orchestrator:** `src/core/AgentLoop.ts` has no native way to attach reusable, declarative rules (e.g. “always be concise”, “use markdown”) to agents; all prompt shaping is done ad‑hoc in each agent markdown, leading to duplicated prompt text and harder rule iteration.

- **File:** `src/agents/AgentRunner.ts` (or create `src/core/RulesetDriver.ts` if preferred).  
- **Change:** Add a small `applyRuleset(prompt: string, ruleset: string[]): string` helper that joins the rule strings (e.g. `"Rule: ${r}"`) and prepends them to the prompt before calling `call_ollama.sh`. Then modify the AgentRunner’s `runAgent` flow to read a `ruleset` field from the agent’s front‑matter (YAML) and invoke the helper. This is 2–3 lines of code (declare the helper, read the field, and wrap the prompt).  

**Effort:** Quick (hours) – only a few lines plus a tiny schema update to the agent markdown files.  

**Risk:** Low – the change is isolated to prompt preparation and falls back gracefully if no ruleset is defined.  

**Verdict: ADOPT** ✅ – Adding a Ruleset driver mirrors Griptape’s proven pattern, eliminates prompt duplication, and enables rapid experimentation with LLM behavior across the orchestrator.

### [omnigent-ai/omnigent](https://github.com/omnigent-ai/omnigent) ⭐ 4442 · Python · Updated: 0 days ago  

**What it does:** Provides a pluggable “policy” system that lets you declare rules (e.g., pause before file writes, cap spend, limit which tools an agent may call) and automatically intercepts agent actions to enforce them.  

**Problem it solves for ai-orchestrator:** `src/core/AgentLoop.ts` currently has no built‑in mechanism to automatically gate risky tool calls (e.g., file‑system writes, shell commands). Adding a policy layer would prevent runaway scripts and give operators a clear “approve → proceed” checkpoint without scattering ad‑hoc checks throughout the code.  

**How to integrate:**  

- **File:** `src/core/AgentLoop.ts` (extend the existing loop)  
- **Change:**  
  1. Import a new `PolicyEngine` class (to be added in `src/core/PolicyEngine.ts`).  
  2. Before each `ToolRunner.execute()` call, invoke `await PolicyEngine.check(action, args)`.  
  3. If the policy returns `{ approved: false, reason: string }`, log the reason and skip the tool; otherwise proceed as before.  

**Effort:** Quick (hours) – adding a small engine module and a two‑line guard per tool execution.  

**Risk:** Low – only adds a conditional branch; existing functionality remains unchanged if policies are empty.  

**Verdict: ADOPT ✅** – the policy pattern is lightweight, aligns with our safety goals, and can be rolled out incrementally.

### [omnigent-ai/omnigent](https://github.com/omnigent-ai/omnigent) ⭐ 4442 · Python · Updated: 0 days ago — OS Sandbox

**What it does:** Uses `bubblewrap` (bwrap) to run each agent harness in an OS‑level sandbox, guaranteeing isolation of file‑system, network, and process resources.  

**Problem it solves for ai-orchestrator:** Our `src/core/ToolRunner.ts` runs arbitrary shell commands (e.g., `run_command`) without any OS‑level confinement, which could be a security liability on shared machines.  

**How to integrate:**  

- **File:** `src/core/ToolRunner.ts` (add a new private method)  
- **Change:**  
  1. Create a helper `sandboxCommand(cmd: string): string` that wraps the incoming command with `bwrap --dev /dev/null --proc /proc --unshare-net --unshare-pid --ro-bind / / <cmd>`.  
  2. In the `run_command` case of the `execute()` switch, replace direct `exec`/`spawn` with `sandboxCommand(cmd)`.  
  3. Guard the wrapper with a runtime check `if (process.platform !== 'linux')` to fall back to plain execution on macOS/Windows.  

**Effort:** Medium (days) – requires testing on Linux containers, handling error propagation, and documenting the optional dependency.  

**Risk:** Medium – introduces a new external binary dependency (`bwrap`); failure to locate it must be gracefully handled to avoid breaking existing workflows.  

**Verdict: WATCH 👀** – the sandboxing concept is valuable but needs careful cross‑platform handling before committing to the codebase.
