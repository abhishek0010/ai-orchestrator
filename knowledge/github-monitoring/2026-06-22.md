### [framerslab/agentos](https://github.com/framerslab/agentos) ⭐ 582 · TypeScript · Updated: 0 days ago

**What it does:** Provides a TypeScript framework where agents can *forge* new tools at runtime, validate them with an LLM judge, and safely load them into a sandboxed VM for later use.

**Problem it solves for ai-orchestrator:** Our current `ToolRunner.ts` only executes a fixed set of built‑in tools (read/write files, run commands). There is no mechanism for agents to **extend** the tool set on the fly, which limits emergent behavior and forces the orchestrator to pre‑declare every possible capability.

**How to integrate:**
- **File:** `src/core/ToolRunner.ts`
- **Change:**  
  1. Add a new `case 'forge_tool'` branch that receives a TypeScript source string and a Zod schema from the planner agent.  
  2. Invoke the existing `AgentRunner` (or create a lightweight helper) to call the LLM judge (`reviewer` role) with the proposed code; treat a non‑empty response as approval.  
  3. On approval, write the source to a temporary file, compile it with `ts-node` (or `esbuild`), and load it via a `node:vm` sandbox (`runInNewContext`) with a 5‑second wall‑clock timeout, mirroring AgentOS’s forge flow.  
  4. Register the loaded function in a new in‑memory map `forgedTools: Record<string, (...args:any[])=>any>` and expose it as a callable tool name for subsequent planner steps.

**Effort:** Quick (hours) – the core logic is a few dozen lines; the rest is wiring into the existing tool‑dispatch path.

**Risk:** Low – the sandboxed VM already exists in our CI test harness, and the feature is isolated to a single new tool type; failure only means the forged tool is rejected.

**Verdict: ADOPT** ✅ – Adding runtime tool forging lets agents create custom utilities on demand, boosting flexibility without major architectural changes.

### [zzet/gortex](https://github.com/zzet/gortex) ⭐ 696 · Go · Updated: 0 days ago

**What it does:** Builds a persistent, multi‑repo code‑intelligence graph (symbols, call‑chains, contracts) and serves it via a fast CLI/MCP/HTTP interface, enabling agents to fetch *only* the exact symbols they need and thus cut token usage dramatically.

**Problem it solves for ai-orchestrator:** `src/core/ContextPruner.ts` currently trims raw file text but has no structural awareness of *which* symbols are actually used. This sometimes forces agents to read large files or whole directories just to locate a single function, inflating token counts.

**How to integrate:**  
- **File:** `src/core/ToolRunner.ts` (add a new tool case)  
- **Change:**  
  1. Add a `symbol_lookup` tool identifier to the `Tool` enum.  
  2. Implement a `case "symbol_lookup":` branch that spawns a thin wrapper script (`scripts/gortex-symbol-lookup.sh`) which calls the Gortex binary (or a lightweight re‑implementation) to query the pre‑built symbol index and returns a JSON payload (e.g. `{symbol: "...", file: "...", span: {start:..., end:...}}`).  
  3. Adjust the planner’s `agents/planner.md` to use this new tool when it needs a definition, reducing the amount of raw source that must be pruned.

**Effort:** Medium (days) – requires adding a new tool entry, writing a modest Bash wrapper, and generating the symbol index (one‑off `gortex index` command).  

**Risk:** Medium – introduces an external binary (the Gortex CLI) and a new dependency on a symbol‑index file; failures must be gracefully degraded to the existing fallback of raw file reads.

**Verdict: WATCH** 👀 – The graph‑based symbol lookup pattern is highly promising for token‑efficient context retrieval, but the full Gortex engine is heavyweight. A modest wrapper can be prototyped now; if it shows measurable token savings, a deeper integration (or a pure‑TS re‑implementation) can be pursued.

### [DeusData/codebase-memory-mcp](https://github.com/DeusData/codebase-memory-mcp) ⭐ 11314 · C · Updated: 0 days ago  

**What it does:** Provides a single‑binary, ultra‑fast code‑intelligence engine that builds a persistent, queryable knowledge‑graph of a codebase (AST, call‑graph, routes, etc.) and exposes rich structural queries via a CLI.  

**Problem it solves for ai‑orchestrator:** The orchestrator currently relies on line‑based diff compression and optional Headroom CLI for token savings. It lacks a *semantic* view of the repository, so agents must fall back to file‑by‑file searching or handcrafted prompts (e.g. `PlannerSession.ts` builds prompts from raw file listings). Adding a structural query layer would let agents ask “which functions call X?” or “what routes exist in this service?” without huge context payloads.  

**How to integrate:**  
- **File:** `src/core/ToolRunner.ts`  
- **Change:**  
  1. Add a new `Tool` enum value `codebase_query`.  
  2. Extend the `execute()` switch with a case that runs the MCP binary (e.g. `codebase-memory-mcp query "<json‑spec>"`) via `execFile`, captures stdout, parses the JSON result, and returns it as a `ToolResult`.  
  3. Document the tool in `src/agents/planner.md` (or a new `agents/mcp_query.md`) so the Planner can request it when a structural query is needed.  

Optionally, expose a thin wrapper script `scripts/run_mcp_query.sh` that checks binary availability (mirroring the pattern of `HeadroomBridge.ts`).  

**Effort:** Medium (≈2 days) – add enum/value, implement the exec path, write JSON parsing helper, and update planner docs.  

**Risk:** Medium – depends on the external MCP binary being installed and compatible across OSes; failure handling can be modelled after existing HeadroomBridge (fallback to empty result).  

**Verdict: ADOPT** ✅ – bringing a proven, high‑performance code‑graph engine into ai‑orchestrator will dramatically reduce token usage for code‑intelligence tasks and give agents a rich structural query language without re‑inventing the wheel.
