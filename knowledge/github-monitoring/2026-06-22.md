### [zzet/gortex](https://github.com/zzet/gortex) ⭐ 697 · Go · Updated: 0 days ago

**What it does:** A high‑performance, language‑agnostic code‑intelligence engine that builds an in‑process graph of symbols, call chains, and contracts to serve AI agents with minimal token overhead.

**Problem it solves for ai-orchestrator:** `ContextPruner.ts` and `HeadroomBridge.ts` currently rely on line‑based pruning and an external headroom CLI; they lack a semantic “graph‑aware” index that can answer “what do I really need?” without reading whole files, leading to sub‑optimal token savings.

**How to integrate:**

- **File:** `src/core/GraphIndex.ts` (new module)
  - **Change:** Add a lightweight in‑memory symbol graph built on the first daemon start using Tree‑Sitter parsers (via the existing `TreeSitter` npm wrapper). Export functions `indexRepo(root:string):Promise<void>` and `lookupSymbols(query:SymbolQuery):SymbolInfo[]`.
- **File:** `src/core/ContextPruner.ts`
  - **Change:** Replace the fallback character‑budget prune with a call to `GraphIndex.lookupSymbols` to fetch only the nodes needed for a given LLM request, then render just those source snippets (plus a small surrounding window). Keep the existing line‑budget fallback for unindexed languages.
- **File:** `src/cli/stats.ts`
  - **Change:** Extend the headroom stats block to also report `graphNodes` and `graphQueries` saved, mimicking Gortex’s “Token savings” CLI output.

**Effort:** Medium (days) – requires adding a new dependency (Tree‑Sitter bindings), wiring the index into the daemon startup, and modest changes to pruning logic.

**Risk:** Medium – introducing a new indexing layer could increase start‑up time and memory footprint; must ensure graceful fallback when parsing fails.

**Verdict:** **ADOPT** ✅ – the graph‑based approach directly addresses the token‑bloat problem in `ContextPruner.ts` and provides a proven token‑saving mechanism (up to 50×) demonstrated by Gortex.
