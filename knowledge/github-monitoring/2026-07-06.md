### [trpc-group/trpc-agent-go](https://github.com/trpc-group/trpc-agent-go) ⭐ 1505 · Go · Updated: 0 days ago

**What it does:** Provides a Go‑native graph‑workflow engine for chaining, parallelising, and conditionally routing LLM agents and tools.

**Problem it solves for ai-orchestrator:** `src/core/Orchestrator.ts` currently runs a single linear pipeline (planner → coder → reviewer). Adding conditional/parallel steps (e.g., “run a second coder if a reviewer flags a problem”) requires a more flexible execution model.

**How to integrate:**

- **File:** `src/core/GraphWorkflow.ts` (new) – define a lightweight `GraphNode` interface, a `Graph` class that holds nodes and edges, and `run(graph: Graph, ctx: ExecutionContext)` that walks the graph, invoking agents via `AgentRunner`.
- **Change:** In `src/core/Orchestrator.ts` add a method `executeGraph(workflow: Graph)` that calls the new `run` helper and returns a `OrchestratorResult`. Update the main loop in `AgentLoop.ts` to construct a simple graph (chain → optional parallel) and invoke `executeGraph` instead of the hard‑coded sequence.

**Effort:** Medium (days) – a few hundred lines of TypeScript, unit tests, and wiring into the existing daemon.

**Risk:** Medium – introducing a new execution model could expose race conditions; needs careful testing of parallel node handling.

**Verdict: ADOPT** ✅ – the graph‑workflow pattern is high‑value (adds conditional/parallel orchestration) and the core logic is straightforward to port to TypeScript without breaking existing files.

### [vitali87/code-graph-rag](https://github.com/vitali87/code-graph-rag) ⭐ 2293 · Python · Updated: 0 days ago

**What it does:** Provides a language‑agnostic code‑base parser (via Tree‑sitter) that builds a knowledge graph of functions, classes, and dependencies for semantic RAG queries.

**Problem it solves for ai-orchestrator:** `ContextPruner.ts` and the planner currently rely on raw file snippets, which can miss structural information (e.g., function boundaries, import graphs). Adding AST‑aware extraction would let agents request “the signature of X” or “all callers of Y” without loading whole files, reducing token usage and improving prompt precision.

**How to integrate:**

- **File:** `src/core/ToolRunner.ts`
- **Change:**
  1. Add a new tool identifier `parse_code_ast` in the tool list front‑matter of the planner agent (e.g., `agents/planner.md`).
  2. Extend `ToolRunner.execute()` with a new case:

     ```ts
     case "parse_code_ast": {
       // delegating to a helper script that runs Tree‑sitter (Python) and returns JSON AST
       const { stdout, error } = await runCommand("python3", ["scripts/parse_ast.py", args.filePath, args.language]);
       return error ? { error } : { result: JSON.parse(stdout) };
     }
     ```

  3. Create a thin wrapper script `scripts/parse_ast.py` (≈30 lines) that invokes Tree‑sitter CLI for the requested language and prints a compact JSON representation of the file’s top‑level symbols (functions, classes, imports).
  4. Update `src/types/index.ts` to add `ParseAstResult` (readonly) if needed.

**Effort:** Medium (days) – requires adding the small Python helper, wiring the new tool, and updating the planner’s prompts to use it.

**Risk:** Medium – introduces a new runtime dependency (Tree‑sitter binaries) and a cross‑language bridge, but the helper is isolated and can fall back to a no‑op if the binary is unavailable.

**Verdict: ADOPT** ✅ – The AST extraction capability directly complements our context‑pruning and planning flow, offering richer, token‑efficient code understanding without breaking existing architecture.
